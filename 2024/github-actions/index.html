GitHub Actions Setup

Posted on 2024-01-23 Edited on 2024-01-24

If you are having issues with the .yml file messing you up, or GitHub keep sending you emails about “!!!Running tasks FAILED” and you see a giant red sea of errors on the Actions page of your repo, it’s likely that you didn’t configure it correctly.

Intro

To simplify the concepts, GitHub Actions is basically a remote virtual machine owned by GitHub that they provided to everyone to run tasks on their code.

Imagine people working on the same project and are constantly changing the code. If you run the entire software manually each time after someone changes one line of code, you will waste loads of man-hours. If you only run the entire software periodically, then it might take a long time to track down who broke the code.

The solution to this is known as CI/CD. For GitHub Actions, after each code change (usually each push), it will automatically follow a predetermined set of commands to:

  • test whether the code looks legit (e.g., pylint)
  • use some fake data to check whether the functions work as predicted (e.g., pytest)
  • check whether people put api keys (similar to passwords) into public code accidentally
  • auto build and publish websites/pages (e.g., to github pages)

Guide

In the flood project, you are given a template pythonapp.yml file and you are requested to change a few lines to run your code properly.

The most difficult part is to set up the github actions virtual machine to make sure it has an environment suitable for running your code. The template file has already set up a Python environment for you and installed some basic packages (e.g., matplotlib). You will be required to make two changes:

  1. (if you used other modules like haversine) install that with pip as well before running the code
  2. under “run deliverable 1/2”, run the respective .py files after you’ve completed them
